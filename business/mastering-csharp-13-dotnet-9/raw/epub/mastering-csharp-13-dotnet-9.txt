1.1 Introduction to C# and .NET: History and Evolution

C# and .NET have played a significant role in shaping modern software development. Whether you're building enterprise applications, cloud services, or cross-platform mobile apps, these technologies provide the tools and structure needed for high-performance, scalable solutions. Understanding how C# and .NET evolved helps you see why they were created, what problems they solve, and how they continue to advance in response to modern software needs.

This section takes you through the history of C# and .NET, from their early days to the latest innovations in C# 13 and .NET 9. By the end, you'll have a strong foundational understanding of where these technologies came from, their guiding principles, and how they shape today's development landscape.

The Origins of C# and .NET

In the late 1990s, software development was undergoing major shifts. Microsoft saw a growing need for a modern, object-oriented, and type-safe programming language that could integrate seamlessly with its ecosystem. The answer was C#, a language designed by Anders Hejlsberg and released in 2000 as part of the first version of the .NET Framework.

.NET was introduced to unify software development under a single ecosystem, offering a framework that would make it easier to build, deploy, and maintain applications. Instead of relying on multiple tools and languages with varying levels of compatibility, developers could now write code in C# and run it reliably across different platforms within the Microsoft ecosystem.

The first version of C# (C# 1.0) was heavily influenced by Java and C++, incorporating modern programming paradigms like object-oriented programming (OOP), garbage collection, and exception handling. Unlike C++, C# removed manual memory management, making it safer and more accessible to developers.

Meanwhile, .NET Framework 1.0 introduced the Common Language Runtime (CLR) and Base Class Library (BCL), providing a runtime environment and a rich set of APIs that made software development more structured and efficient.

Key Milestones in C# and .NET Evolution

As the needs of developers grew, Microsoft consistently improved C# and .NET with new features, better performance, and expanded support for modern development trends. Here’s how both technologies evolved:

C# Language Evolution

C# 2.0 (2005) – Introduced generics, iterators, nullable types, and partial classes, significantly improving code reuse and maintainability. 
C# 3.0 (2007) – Brought in LINQ (Language Integrated Query), lambda expressions, and automatic properties, making data manipulation more powerful and expressive. 
C# 4.0 (2010) – Introduced dynamic typing, named/optional parameters, and better COM interoperability, providing more flexibility. 
C# 5.0 (2012) – Revolutionized asynchronous programming with async and await, making it easier to write non-blocking, scalable applications. 
C# 6.0 (2015) – Brought expression-bodied members, interpolated strings, and null-conditional operators for more concise and readable code. 
C# 7.0-7.3 (2017-2018) – Expanded support for tuples, pattern matching, local functions, and performance optimizations. 
C# 8.0 (2019) – Introduced nullable reference types and async streams, enhancing code safety and performance. 
C# 9.0 (2020) – Focused on immutability with records and simplified object initialization with init-only properties. 
C# 10.0 (2021) – Improved global using directives, file-scoped namespaces, and record struct types, making code more maintainable. 
C# 11.0 (2022) – Enhanced raw string literals, generic math support, and introduced ref fields in structs. 
C# 12.0 (2023) – Introduced primary constructors for all classes, aliasing any type, and improved default lambda parameters. 
C# 13.0 (2024-2025) – Focuses on performance, AI-assisted development, improved async programming, and deeper integration with .NET 9. 

.NET Framework to .NET 9: The Cross-Platform Revolution

While C# was evolving, the .NET ecosystem underwent its own transformation.

● .NET Framework 1.0 to 4.8 (2002-2019) – Initially designed for Windows applications, .NET Framework powered enterprise applications, but its Windows-centric nature limited cross-platform use.

● .NET Core 1.0 (2016) – Microsoft rebuilt .NET from the ground up to be open-source, lightweight, and cross-platform. .NET Core introduced a modular design and high-performance runtime.

● .NET 5 (2020) – Unified .NET Core and .NET Framework into a single platform, setting the foundation for future versions.

● .NET 6 (2021) – First LTS (Long-Term Support) release, introducing minimal APIs, hot reload, and performance optimizations.

● .NET 7 (2022) – Enhanced Cloud-Native support, performance improvements, and deep Blazor integration.

● .NET 8 (2023) – Expanded AI capabilities, ASP.NET Core enhancements, and better container support.

● .NET 9 (2024-2025) – Focuses on performance, memory optimizations, and developer productivity, further refining Blazor, ASP.NET Core, and EF Core.

Why C# 13 and .NET 9 Matter Today

With C# 13 and .NET 9, Microsoft continues to push the boundaries of what developers can achieve. Some of the most exciting advancements include:

● Even Faster Execution – .NET 9 continues optimizing runtime performance, making applications more efficient.

● Improved AI and Cloud Integration – Enhanced support for AI-driven applications and tighter cloud service integration.

● Blazor Enhancements – Blazor now supports even more interactive, high-performance web applications with improved developer experience.

● Security and Stability – Stronger encryption, enhanced authentication features, and better debugging tools.

● Seamless Cross-Platform Deployment – .NET 9 makes it easier to develop once and deploy anywhere: Windows, Linux, macOS, cloud, or mobile.

Real-World Examples of C# and .NET in Action

Enterprise Applications – Large companies use C# and .NET for business-critical applications like banking systems, CRM software, and HR management tools. 
Cloud-Based Services – .NET powers cloud platforms like Azure Functions, AWS Lambda, and Google Cloud Run. 
Game Development – Unity, one of the most popular game engines, is built on C#. 
Web Applications – ASP.NET Core is used in high-traffic websites and web services for e-commerce, SaaS products, and APIs. 
AI and Machine Learning – .NET integrates with ML.NET and AI frameworks for predictive analytics, chatbots, and automation.

2.1 Variables, Data Types, and Operators

Programming revolves around storing, manipulating, and processing data. Whether you're developing a small utility script or a large enterprise application, you need to store values, perform calculations, and control logic flow. In C# 13, this is done using variables, data types, and operators.

Understanding these fundamental concepts is crucial because they directly affect how efficiently your application runs, how memory is managed, and how errors are handled. This section breaks down these concepts with clear explanations and practical examples so that you can confidently apply them in real-world applications.

2.1.1 What Are Variables?

A variable is a named storage location in memory that holds a value. Think of it as a container where you can store and retrieve data. In C#, every variable has a type, which defines the kind of data it can store, such as numbers, text, or boolean values.

Declaring a Variable in C#

In C#, a variable is declared using the following syntax:

<data_type> variableName = value;

For example:

int age = 30;

string name = "John Doe";

double price = 9.99;

Here’s what happens in memory:

● age is an integer that holds the value 30.

● name is a string that holds "John Doe".

● price is a double (floating-point number) that holds 9.99.

Variables allow you to store values and reference them later in your program. You can modify their values at any time unless they are declared as constants.

2.1.2 Understanding Data Types

Every variable in C# has a data type that defines what kind of value it can store. C# 13 provides two main categories of data types:

1. Value Types

These store data directly in memory. Examples include:

● int (integer numbers)

● double, float (floating-point numbers)

● char (single character)

● bool (true or false)

● struct (user-defined structure)

Example of using value types:

int count = 10;

double temperature = 36.5;

bool isActive = true;

char grade = 'A';

2. Reference Types

These store references to memory locations instead of actual data. Examples include:

● string (text)

● object (base type for all data types)

● class (user-defined blueprint for objects)

● dynamic (runtime-determined type)

Example of using reference types:

string message = "Hello, World!";

object obj = new object();

dynamic variable = 42; // Can change to any type

2.1.3 Implicit and Explicit Variable Declaration

C# allows both explicit and implicit declarations:

Explicit Declaration

You specify the exact data type:

int number = 25;

string text = "C# Programming";

Implicit Declaration (var)

The compiler infers the type at compile-time:

var count = 10; // Inferred as int

var title = "C#"; // Inferred as string

var price = 4.99; // Inferred as double

You cannot use var without initialization because the compiler needs a value to determine the type:

var x; // Error: Implicitly typed variables must be initialized

When to Use var?

● Use var when the type is obvious from the right-hand value.

● Avoid var when clarity is reduced.

2.1.4 Constants and Read-Only Variables

If you need a fixed value that cannot be changed, use const or readonly:

Constant (const)

The value must be assigned at declaration and cannot change:

const double Pi = 3.14159;

Read-Only (readonly)

Can be assigned only once, typically in the constructor of a class:

readonly int maxUsers;

public Program()

{

maxUsers = 100;

}

Use const for values known at compile-time, and readonly for values assigned at runtime.

2.1.5 Operators in C#

Operators allow you to perform calculations, comparisons, and logic-based operations.

1. Arithmetic Operators

Used for mathematical calculations:

int a = 10;

int b = 5;

int sum = a + b; // Addition (15)

int diff = a - b; // Subtraction (5)

int prod = a * b; // Multiplication (50)

int quotient = a / b; // Division (2)

int remainder = a % b; // Modulus (0)

2. Comparison Operators

Used for comparing values:

bool isEqual = (a == b); // false

bool isNotEqual = (a != b); // true

bool isGreater = (a > b); // true

bool isLess = (a < b); // false

3. Logical Operators

Used to perform logical operations:

bool x = true;

bool y = false;

bool andResult = x && y; // false

bool orResult = x || y; // true

bool notResult = !x; // false

4. Assignment Operators

Used to assign values:

int num = 10;

num += 5; // Same as num = num + 5 (15)

num -= 3; // Same as num = num - 3 (12)

num *= 2; // Same as num = num * 2 (24)

5. Null-Coalescing Operators (?? and ??=)

C# 13 allows safer handling of null values:

string? user = null;

string displayName = user ?? "Guest"; // If user is null, use "Guest"

user ??= "DefaultUser"; // Assign "DefaultUser" only if user is null

2.1.6 Practical Example: Simple Calculator

Let’s apply what we’ve learned by creating a basic calculator in C#:

using System;

class Calculator

{

static void Main()

{

Console.Write("Enter first number: ");

2.3.1 What Are Exceptions?

2.4 Introduction to Asynchronous Programming with async and await

Modern applications often require handling multiple tasks simultaneously to deliver a smooth and responsive user experience. For instance, downloading data from the internet, reading a file, or querying a database should not block the rest of your application. This is where asynchronous programming comes into play.

In C#, asynchronous programming allows you to perform tasks without freezing the main thread. The introduction of async and await in C# 5 revolutionized asynchronous programming, making it more intuitive and accessible. In C# 13, these features have been further optimized for performance and usability.

This section will provide a comprehensive explanation of how async and await work, how to use them effectively, and how they apply to real-world scenarios.

2.4.1 What is Asynchronous Programming?

Asynchronous programming allows you to execute long-running operations without blocking the main thread. In applications like desktop apps, web servers, or mobile apps, blocking the main thread can lead to a poor user experience, such as unresponsive interfaces or delays in processing.

Synchronous vs Asynchronous Execution

● Synchronous Execution: Tasks are executed one after another. The next task doesn’t begin until the current one finishes.

● Asynchronous Execution: Tasks can run independently, allowing the program to continue executing other code while waiting for a long-running task to complete.

2.4.2 The async and await Keywords

The keywords async and await make asynchronous programming easy and readable in C#.

The async Keyword

● The async modifier marks a method as asynchronous.

● An async method can contain await expressions.

● It always returns one of the following types: Task: For methods that perform operations but don’t return a result. Task<T>: For methods that return a value. void: For asynchronous event handlers (rarely used elsewhere).

The await Keyword

● The await keyword pauses the execution of the current method until the awaited task completes.

● While the method is paused, control is returned to the calling method, allowing other code to execute.

2.4.3 Basic Example of async and await

Here’s a simple example to demonstrate asynchronous programming in C#:

using System;

using System.Threading.Tasks;

class Program

{

static async Task Main()

{

Console.WriteLine("Starting task...");

string result = await PerformTaskAsync();

Console.WriteLine($"Task completed: {result}");

}

static async Task<string> PerformTaskAsync()

{

await Task.Delay(3000); // Simulates a 3-second delay

return "Task is done!";

}

}

Explanation:

The PerformTaskAsync method is marked as async, meaning it can run asynchronously. 
The Task.Delay(3000) simulates a time-consuming operation (like downloading data) that takes 3 seconds. 
The await keyword in Main pauses execution until PerformTaskAsync completes. During this time, the main thread remains free to handle other tasks. 

2.4.4 Real-World Scenario: Fetching Data from an API

Let’s take a practical example: fetching data from a web API asynchronously.

using System;

using System.Net.Http;

using System.Threading.Tasks;

class Program

{

static async Task Main()

{

Console.WriteLine("Fetching data...");

string data = await FetchDataFromApiAsync("https://jsonplaceholder.typicode.com/posts/1");

Console.WriteLine($"Data received: {data}");

}

static async Task<string> FetchDataFromApiAsync(string url)

{

using HttpClient client = new HttpClient();

HttpResponseMessage response = await client.GetAsync(url);

response.EnsureSuccessStatusCode(); // Throws an exception if the HTTP request failed

string responseData = await response.Content.ReadAsStringAsync();

return responseData;

}

}

How This Works:

HttpClient is used to make an HTTP GET request. 
The GetAsync method fetches data asynchronously, so the main thread remains free. 
The await response.Content.ReadAsStringAsync() reads the response data without blocking the application. 

This approach is particularly useful for web applications, where blocking the thread could lead to a poor user experience.

When working with asynchronous code, you still need to handle exceptions effectively. Use a try-catch block to catch errors in async methods.

2.4.6 Running Multiple Tasks Concurrently

You can run multiple asynchronous tasks in parallel using Task.WhenAll.

Example: Running Tasks Concurrently

using System;

using System.Threading.Tasks;

class Program

{

static async Task Main()

{

Task<string> task1 = PerformTaskAsync("Task 1", 2000);

Task<string> task2 = PerformTaskAsync("Task 2", 3000);

string[] results = await Task.WhenAll(task1, task2);

foreach (string result in results)

{

Console.WriteLine(result);

}

}

static async Task<string> PerformTaskAsync(string taskName, int delay)

{

await Task.Delay(delay);

return $"{taskName} completed after {delay} ms";

}

}

Explanation:

● Two tasks are started simultaneously (task1 and task2).

● Task.WhenAll waits for both tasks to complete and returns their results.

● This approach improves performance when multiple independent tasks need to run.

2.4.7 Best Practices for Asynchronous Programming

Use Async All the Way
When a method is asynchronous, every method that calls it should also be asynchronous.

Avoid Blocking Code
Do not use .Result or .Wait() to block asynchronous code, as this can lead to deadlocks.

Leverage Task.WhenAll
Use Task.WhenAll or Task.WhenAny for handling multiple tasks efficiently.

Handle Exceptions Gracefully
Always wrap your asynchronous methods in try-catch blocks to handle potential errors.

Use Cancellation Tokens
Allow users or processes to cancel long-running tasks with cancellation tokens.

4.1.1 Understanding Records in C# 13

● Concise syntax (Eliminates boilerplate code).

● Immutability by default (Ensures safer data handling).

4.2.2 Working with Events in C#

Events in C# are built on delegates and provide a structured way for objects to notify subscribers when something important happens.

How Events Work

● An event publisher defines an event.

● Subscribers (other parts of the code) register methods to listen for the event.

● When the event is triggered, all registered methods execute.

Declaring and Raising an Event

using System;

class Alarm

{

// Declaring an event using a delegate

public delegate void AlarmHandler(string location);

public event AlarmHandler AlarmTriggered;

public void TriggerAlarm(string location)

{

Console.WriteLine("Alarm triggered!");

AlarmTriggered?.Invoke(location); // Raising the event

}

}

class SecuritySystem

{

static void Alert(string location)

{

Console.WriteLine($"Security alert! Intrusion detected at {location}.");

}

static void Main()

{

Alarm alarm = new Alarm();

// Subscribing to the event

alarm.AlarmTriggered += Alert;

// Triggering the event

alarm.TriggerAlarm("Main Gate");

}

}

Output:

Alarm triggered!

Security alert! Intrusion detected at Main Gate.

Concepts Demonstrated:

● AlarmTriggered is an event based on the AlarmHandler delegate.

● += operator subscribes a method (Alert) to the event.

● ?.Invoke() safely raises the event if there are subscribers.

4.4 Introduction to Nullable Reference Types and Best Practices

Handling null values correctly is one of the most challenging aspects of programming. A null reference exception is a common runtime error that occurs when trying to access members of a null object. Nullable reference types (NRTs) were introduced in C# to minimize this issue, improving type safety and helping developers write more reliable code.

With nullable reference types, the compiler provides warnings and analysis to prevent null reference issues at compile time instead of letting them cause unexpected failures at runtime. This section explores how nullable reference types work, their benefits, and best practices for using them effectively in C# 13.

4.4.1 What Are Nullable Reference Types?

By default, reference types (e.g., string, object) can be assigned null. While this is sometimes necessary, it often leads to null reference exceptions, which are difficult to debug. Nullable reference types provide a way to explicitly declare whether a reference type variable can or cannot be null, allowing the compiler to enforce correctness.

Enabling Nullable Reference Types

Nullable reference types are enabled by default in new projects. However, for existing projects, you may need to manually enable them in the .csproj file:

<PropertyGroup>

<Nullable>enable</Nullable>

</PropertyGroup>

Alternatively, you can enable it within a C# file:

#nullable enable

This tells the compiler to enforce null safety rules on reference types.

4.4.2 Nullable vs. Non-Nullable Reference Types

Non-Nullable Reference Types (Default Behavior)

When nullable reference types are enabled, all reference type variables are considered non-nullable by default.

string message = "Hello, C# 13!";

message = null; // Compiler Warning: Cannot assign null to a non-nullable variable

Nullable Reference Types (? Modifier)

To explicitly allow a variable to be null, you append a ? to the type.

string? nullableMessage = null; // No compiler warning

Since nullableMessage can be null, the compiler requires a null check before accessing its members.

if (nullableMessage != null)

{

Console.WriteLine(nullableMessage.Length); // Safe access

}

Key Takeaway:

● Use string when you guarantee the variable will never be null.

● Use string? when null values are expected and should be handled.

4.4.3 Compiler Warnings and Static Analysis

The biggest advantage of nullable reference types is that the compiler warns you when a potential null reference exists.

void PrintMessage(string? msg)

{

Console.WriteLine(msg.Length); // Compiler Warning: Possible null reference

}

Fixing Potential Null Dereference with Null Checking

void PrintMessage(string? msg)

{

if (msg != null)

{

Console.WriteLine(msg.Length); // Safe

}

else

{

Console.WriteLine("Message is null.");

}

}

By requiring explicit null checks, nullable reference types eliminate many null-related runtime errors.

4.4.4 Using the Null Conditional (?.) and Null Coalescing (??) Operators

To simplify null handling, C# provides operators like ?. (null conditional) and ?? (null coalescing).

Using ?. to Avoid Null Dereference Errors

Instead of writing an explicit if check, you can use ?. to safely access properties of potentially null objects.

string? nullableText = null;

Console.WriteLine(nullableText?.Length); // No error, outputs: (null)

Using ?? to Provide a Default Value

The null coalescing (??) operator returns a default value when a variable is null.

string? name = null;

Console.WriteLine(name ?? "No name provided"); // Output: No name provided

These operators simplify code while maintaining null safety.

4.4.5 Null Forgiving Operator (!)

In rare cases where you're sure a variable isn't null but the compiler still warns you, you can use the null forgiving (!) operator.

string? nullableName = GetName();

Console.WriteLine(nullableName!.Length); // No compiler warning

However, using ! should be rare—it tells the compiler to ignore null safety checks, which can lead to runtime errors if misused.

4.4.6 Applying Nullable Reference Types in Real-World Scenarios

Scenario 1: Handling Database Queries

When working with databases, some columns might allow null values. Nullable reference types ensure you handle null results properly.

public class UserRepository

{

public string? GetUserById(int id)

{

// Simulating a database lookup that might return null

return id == 1 ? "Alice" : null;

}

}

public class Program

{

public static void Main()

{

UserRepository repo = new UserRepository();

string? user = repo.GetUserById(2);

Console.WriteLine(user ?? "User not found"); // Output: User not found

}

}

This approach prevents null reference exceptions in database access.

Scenario 2: Using NRTs in API Models

In API development, some fields might be optional. Nullable reference types make this explicit.

public class User

{

public string Name { get; set; } // Required field

public string? Address { get; set; } // Optional field

}

This helps document API expectations clearly and prevents unintentional null-related issues.

Scenario 3: Ensuring Safe Dependency Injection

Dependency injection (DI) frameworks often inject dependencies into classes. Nullable reference types prevent missing dependencies.

public class EmailService

{

private readonly SmtpClient _smtpClient;

public EmailService(SmtpClient smtpClient)

{

_smtpClient = smtpClient ?? throw new ArgumentNullException(nameof(smtpClient));

}

}

This ensures that _smtpClient is never null, preventing misconfigurations.

4.4.7 Best Practices for Using Nullable Reference Types

Enable nullable reference types (#nullable enable) in all new projects to prevent silent null errors. 
Use ? only when a value is explicitly allowed to be null—otherwise, keep reference types non-nullable. 
Use null-checking techniques (if conditions, ?., ??) to safely handle nullable variables. 
Avoid excessive use of the null-forgiving (!) operator, as it can lead to runtime errors. 
Annotate API models and database entities properly to indicate optional and required fields clearly. 
Refactor legacy code gradually by enabling nullable reference types and fixing warnings step by step. 

4.4.8 Conclusion

Nullable reference types are a powerful tool in C# 13 that significantly reduce the likelihood of null reference exceptions. By enforcing null safety at compile-time, they help developers write more robust, maintainable, and predictable code.

● They eliminate unintended null reference issues.

● They improve code clarity by distinguishing nullable and non-nullable types.

● They encourage safe coding practices using compiler warnings.

By applying best practices and leveraging null safety features, developers can write cleaner, safer, and more reliable applications in C# 13.

5.2.1 Understanding Entities and Relationships in EF Core

What is an Entity?

An entity in EF Core is a C# class that represents a table in a database. Each entity has properties that map to table columns.

5.2.2 One-to-One Relationship in EF Core

A one-to-one relationship means that a record in one table is related to exactly one record in another table.

Example: One-to-One Relationship Between User and UserProfile

public class User

{

public int Id { get; set; }

public string Name { get; set; } = string.Empty;

// Navigation Property

public UserProfile? Profile { get; set; }

}

public class UserProfile

{

public int Id { get; set; }

public string Address { get; set; } = string.Empty;

// Foreign Key

public int UserId { get; set; }

public User User { get; set; } = null!;

}

Configuring One-to-One Relationship in DbContext

protected override void OnModelCreating(ModelBuilder modelBuilder)

modelBuilder.Entity<User>()

.HasOne(u => u.Profile)

.WithOne(p => p.User)

.HasForeignKey<UserProfile>(p => p.UserId);

Explanation:

● User has one UserProfile, and UserProfile has one User.

● Foreign key (UserId) in UserProfile ensures referential integrity.

● The navigation properties allow easy access to related data.

5.2.3 One-to-Many Relationship in EF Core

A one-to-many relationship means that one record in a parent table can be associated with multiple records in a child table.

Example: One-to-Many Relationship Between Category and Product

public class Category

{

public class Product

{

public int Id { get; set; }

// Foreign Key

protected override void OnModelCreating(ModelBuilder modelBuilder)

{

modelBuilder.Entity<Product>()

.HasOne(p => p.Category)

.WithMany(c => c.Products)

.HasForeignKey(p => p.CategoryId);

Explanation:

● A Category can have many Products, but each Product belongs to only one Category.

● The CategoryId foreign key in Product ensures proper linkage.

● The List<Product> navigation property in Category allows easy access to all products in a category.

5.4 Performance Optimization Techniques for EF Core Queries

Efficient database queries are essential for maintaining scalability, responsiveness, and resource efficiency in applications. While Entity Framework Core 9 (EF Core 9) abstracts many complexities of database interactions, poorly optimized queries can negatively impact performance, leading to slow execution times, excessive memory consumption, and unnecessary database load.

This section focuses on practical techniques to optimize EF Core queries, ensuring that database interactions remain fast and efficient.

5.4.1 Understanding Performance Bottlenecks in EF Core

Before optimizing queries, it's crucial to identify where performance issues occur. Common bottlenecks include:

● Unnecessary queries: Fetching more data than needed.

● Too many queries (N+1 problem): Running multiple queries instead of a single optimized query.

● Inefficient data retrieval: Loading entire entities when only a few fields are needed.

● Lack of indexing: Querying unindexed columns slows down lookups.

● Excessive database round-trips: Making frequent queries instead of batch processing.

5.4.2 Optimizing Data Retrieval

Fetching Only Required Columns with Select()

Many developers make the mistake of retrieving entire entities when only a few fields are needed. This increases memory usage and execution time.

Example: Fetching Only Necessary Fields

var productSummaries = context.Products

.Select(p => new { p.Name, p.Price })

.ToList();

foreach (var product in productSummaries)

{

Console.WriteLine($"{product.Name} - ${product.Price}");

}

Why is this better?

● Reduces memory allocation by fetching only needed fields.

● Optimizes query execution time.

SQL Generated:

SELECT Name, Price FROM Products;

Using AsNoTracking() for Read-Only Queries

By default, EF Core tracks entities in memory. If a query is read-only, tracking is unnecessary and adds overhead.

Example: Disabling Tracking for Performance

var customers = context.Customers

.AsNoTracking()

.ToList();

Benefits:

● Improves performance for large result sets by disabling change tracking.

● Reduces memory usage.

Use Case:

● Ideal for displaying data in UI elements like reports, dashboards, or read-only pages.

5.4.3 Reducing the Number of Queries (Solving the N+1 Problem)

The N+1 problem occurs when a parent entity is retrieved, and then separate queries fetch each child entity individually, leading to too many queries hitting the database.

Example of the N+1 Problem (Inefficient Querying)

var categories = context.Categories.ToList();

foreach (var category in categories)

{

var products = context.Products.Where(p => p.CategoryId == category.Id).ToList();

}

If there are 100 categories, this results in 101 queries (1 for categories, 100 for products), creating a performance bottleneck.

Solution: Use Eager Loading with .Include()

var categoriesWithProducts = context.Categories

.Include(c => c.Products)

.ToList();

Why is this better?

● Reduces multiple queries into a single optimized query.

● Fetches categories and their products in one database request.

SQL Generated:

SELECT * FROM Categories

JOIN Products ON Categories.Id = Products.CategoryId;

5.4.4 Optimizing Filtering and Indexing

Using Efficient Filtering to Reduce Data Load

Fetching only relevant records is crucial for performance.

Example: Retrieving Only Active Users

var activeUsers = context.Users

.Where(u => u.IsActive)

.ToList();

Ensuring Indexing on Frequently Queried Columns

Indexes speed up data retrieval for common queries. If filtering by Email frequently, ensure an index exists.

SQL to Add Index Manually:

CREATE INDEX IX_Users_Email ON Users (Email);

Best Practice:

● Always index columns used in WHERE and JOIN conditions.

● Avoid indexing low-cardinality columns (e.g., IsActive with only true/false values).

5.4.5 Batch Processing and Bulk Operations

Avoiding Multiple Saves with SaveChanges()

Calling SaveChanges() multiple times increases database round-trips.

Inefficient Approach (Multiple Queries):

foreach (var order in orders)

{

context.Orders.Add(order);

context.SaveChanges(); // BAD: Calls the database each time

}

Optimized Approach (Single Batch Operation):

context.Orders.AddRange(orders);

context.SaveChanges(); // Executes a single batch insert

Why is this better?

● Reduces database round-trips.

● Improves efficiency for bulk inserts and updates.

5.4.6 Using Stored Procedures for Complex Queries

Sometimes, complicated queries run faster as stored procedures in the database.

Example: Calling a Stored Procedure in EF Core

var topProducts = context.Products

.FromSqlRaw("EXEC GetTopSellingProducts")

.ToList();

Benefits:

● Optimized execution plans.

● Reduces network latency.

5.4.7 Caching Query Results for Frequent Queries

Repeated queries can be optimized with caching to avoid redundant database hits.

Example: Using Memory Cache to Store Query Results

Why is this better?

5.4.8 Asynchronous Queries for Improved Responsiveness

Synchronous queries block execution, causing performance issues in high-traffic applications.

Example: Fetching Data Asynchronously

var users = await context.Users.ToListAsync();

Benefits:

● Frees up thread resources.

● Improves scalability in web applications.

5.4.9 Best Practices for EF Core Performance Optimization

Use .Select() to fetch only required fields, reducing memory usage. 
Disable tracking (AsNoTracking()) for read-only queries. 
Use .Include() wisely to prevent the N+1 problem. 
Index frequently queried columns for faster lookups.

6.3 Implementing Middleware for Custom HTTP Pipelines in ASP.NET Core 9

Handling HTTP requests efficiently is a critical aspect of web application performance, security, and logging. ASP.NET Core 9 uses a middleware-based pipeline to process incoming requests and generate responses. This section explains how middleware works, how to implement custom middleware, and how to optimize HTTP request handling using ASP.NET Core 9.

6.3.1 Understanding Middleware in ASP.NET Core 9

What is Middleware?

Middleware is software logic that processes HTTP requests and responses in a pipeline. Each middleware component can:

● Modify the request before it reaches the next component.

● Generate a response directly, stopping further processing.

● Pass the request to the next middleware for additional processing.

Middleware is executed in the order it is registered, making pipeline configuration critical for performance and security.

Example: Default Middleware Pipeline in ASP.NET Core 9

When creating a new ASP.NET Core 9 application, the default middleware pipeline might look like this:

var builder = WebApplication.CreateBuilder(args);

app.UseAuthorization();

app.MapGet("/", () => "Hello, ASP.NET Core 9!");

app.Run();

Explanation:

UseRouting() → Determines the correct route for the request. 
UseAuthorization() → Enforces authentication and authorization policies. 
MapGet("/") → Handles the HTTP request and returns a response. 

The request flows through each middleware component in sequence.

6.3.2 Implementing Custom Middleware in ASP.NET Core 9

Sometimes, built-in middleware isn't enough. Custom middleware allows developers to intercept and manipulate requests to handle logging, error handling, security headers, and performance optimizations.

Creating a Simple Custom Middleware

using Microsoft.AspNetCore.Http;

using System.Diagnostics;

using System.Threading.Tasks;

public class RequestTimingMiddleware

{

private readonly RequestDelegate _next;

public RequestTimingMiddleware(RequestDelegate next)

{

_next = next;

}

public async Task InvokeAsync(HttpContext context)

{

var stopwatch = Stopwatch.StartNew();

await _next(context); // Call the next middleware in the pipeline

stopwatch.Stop();

Console.WriteLine($"Request {context.Request.Method} {context.Request.Path} took {stopwatch.ElapsedMilliseconds} ms");

}

Registering the Middleware in Program.cs

var builder = WebApplication.CreateBuilder(args);

var app = builder.Build();

app.UseMiddleware<RequestTimingMiddleware>();

app.MapGet("/", () => "Hello, World!");

app.Run();

How It Works:

● The middleware starts a stopwatch before passing the request to the next component.

● When the request is completed, it logs the execution time.

● This helps monitor performance by measuring response times.

6.3.3 Using Use() and Run() for Inline Middleware

ASP.NET Core 9 provides two methods for defining middleware inline:

Use() → Calls the next middleware in the pipeline. 
Run() → Ends the request pipeline and returns a response. 

Example: Logging Middleware Using Use()

var builder = WebApplication.CreateBuilder(args);

var app = builder.Build();

app.Use(async (context, next) =>

{

});

app.Run(async context =>

{

});

Why Use Run()?

● Stops request execution immediately, useful for maintenance pages or security filters.

6.3.4 Handling Exceptions with Global Error Middleware

Instead of try-catch blocks inside controllers, it's more efficient to centralize error handling in middleware.

Creating an Error Handling Middleware

using Microsoft.AspNetCore.Http;

using System.Net;

using System.Text.Json;

using System.Threading.Tasks;

public class ExceptionHandlingMiddleware

{

private readonly RequestDelegate _next;

public ExceptionHandlingMiddleware(RequestDelegate next)

{

_next = next;

}

{

try

{

await _next(context);

}

catch (Exception ex)

{

context.Response.StatusCode = (int)HttpStatusCode.InternalServerError;

context.Response.ContentType = "application/json";

}

Why Use This?

● Centralizes error handling instead of handling exceptions in multiple places.

● Prevents application crashes by responding with a structured error message.

● Implement security headers to harden applications against attacks.

6.4 Authentication and Authorization in ASP.NET Core 9

Security is a fundamental aspect of modern web applications. Authentication verifies a user's identity, while authorization determines what actions they can perform. 6.4.1 Understanding Authentication and Authorization

6.4.3 Generating JWT Tokens for User Authentication

To generate a JWT token, create a function that issues a signed JWT token when a user logs in.

Example: Creating a JWT Token Generator

using System.IdentityModel.Tokens.Jwt;

using System.Security.Claims;

using Microsoft.IdentityModel.Tokens;

using System.Text;

public class TokenService

{

private const string SecretKey = "YourSecretKey1234567890"; // Secure this key properly

public static string GenerateToken(string username, string role)

{

var tokenHandler = new JwtSecurityTokenHandler();

var key = Encoding.UTF8.GetBytes(SecretKey);

var claims = new List<Claim>

{

new Claim(ClaimTypes.Name, username),

new Claim(ClaimTypes.Role, role)

};

var tokenDescriptor = new SecurityTokenDescriptor

{

Subject = new ClaimsIdentity(claims),

Expires = DateTime.UtcNow.AddHours(1), // Token expiry

Issuer = "https://yourdomain.com",

Audience = "https://yourdomain.com",

SigningCredentials = new SigningCredentials(new SymmetricSecurityKey(key), SecurityAlgorithms.HmacSha256Signature)

};

var token = tokenHandler.CreateToken(tokenDescriptor);

return tokenHandler.WriteToken(token);

}

Example: Issuing a Token on User Login

app.MapPost("/login", (UserLogin request) =>

{

if (request.Username == "admin" && request.Password == "password")

{

var token = TokenService.GenerateToken(request.Username, "Admin");

return Results.Ok(new { Token = token });

}

return Results.Unauthorized();

});

public record UserLogin(string Username, string Password);

6.4.4 Securing API Endpoints with Authorization

Protecting Endpoints with JWT Authentication

To restrict access to authenticated users, apply the [Authorize] attribute:

app.MapGet("/secure-data", () => "This is protected data!")

.RequireAuthorization();

To restrict access to users with specific roles:

app.MapGet("/admin", () => "Admin Dashboard")

6.4.5 Implementing Policy-Based Authorization

Sometimes, custom rules (e.g., "Only users with verified emails can access") are required. ASP.NET Core supports policy-based authorization.

Defining a Custom Authorization Policy

Modify Program.cs to add a custom verified email policy:

builder.Services.AddAuthorization(options =>

{

options.AddPolicy("VerifiedEmail", policy =>

policy.RequireClaim("EmailVerified", "true"));

});

Applying the Policy to an Endpoint

app.MapGet("/verified-users", () => "Verified Users Only")

.RequireAuthorization("VerifiedEmail");

Issuing a JWT Token with Custom Claims

Modify TokenService.GenerateToken() to add an email verification claim:

var claims = new List<Claim>

{

new Claim(ClaimTypes.Name, username),

new Claim(ClaimTypes.Role, role),

new Claim("EmailVerified", "true") // Custom claim

};

Now, only users with the EmailVerified claim can access /verified-users.

6.4.6 Best Practices for Secure Authentication and Authorization

Implement refresh tokens – Avoid frequent logins by issuing refresh tokens. 
Log authentication attempts – Monitor login failures to detect suspicious activity. 
Validate input data – Prevent brute force attacks with rate-limiting and validation. 
Enforce least privilege – Grant users the minimum required access using roles and policies.

ASP.NET Core 9 provides a robust and flexible authentication and authorization system, allowing developers to secure APIs efficiently. By using JWT tokens, role-based access, and policy-based authorization, applications can ensure only authorized users access sensitive data.

Chapter 7: Creating Interactive Web Applications with Blazor

Blazor is a modern web framework that enables developers to build interactive and dynamic web applications using C# and .NET, eliminating the need for JavaScript in many scenarios. 7.1 Introduction to Blazor: Server vs. WebAssembly

Modern web applications demand interactivity, responsiveness, and rich user experiences. Traditionally, JavaScript frameworks like React, Angular, and Vue.js dominated frontend development. Blazor, a framework in ASP.NET Core, enables developers to build feature-rich, interactive web applications using C# and .NET, reducing reliance on JavaScript.

Blazor comes in two primary hosting models:

Blazor Server – Runs on the server and updates the UI using a real-time SignalR connection. 
Blazor WebAssembly (WASM) – Runs entirely in the browser, using WebAssembly (WASM) to execute C# code. 

Both models share the same component-based architecture, allowing developers to choose the approach best suited to their needs.

7.1.1 What is Blazor?

Blazor is a component-based UI framework that allows developers to build single-page applications (SPAs) using C# and Razor syntax. Unlike traditional .NET applications, Blazor enables client-side execution of .NET code, making it possible to write both frontend and backend code in C#.

Key Features of Blazor

● Runs in the Browser (via WebAssembly) or on the Server

● Uses C# Instead of JavaScript

● Component-Based Architecture

● Supports Dependency Injection and Routing

● Works with Existing .NET Libraries

● Built-In State Management

Blazor simplifies web development by allowing full-stack development in .NET, removing the need to switch between C# for backend logic and JavaScript for UI interactions.

7.1.2 Understanding Blazor Hosting Models: Server vs. WebAssembly

Blazor applications can run in two different hosting models, each with distinct advantages and trade-offs.

Blazor Server: Running on the Server with SignalR

Blazor Server applications execute on the server, while the UI updates in real-time using a SignalR WebSocket connection.

How Blazor Server Works

The browser loads an initial page from the server. 
All UI logic executes on the server, not in the browser. 
When a user interacts with the UI (e.g., clicks a button), an event is sent to the server via SignalR. 
The server processes the event, updates the UI, and sends back the updated UI state to the browser. 
The browser applies the UI updates without reloading the page. 

Setting Up a Blazor Server App

Create a Blazor Server application using the .NET CLI:

How This Works in Blazor Server:

● The counter value is updated on the server.

● The updated UI is sent back to the client over SignalR.

● No full-page refresh is required.

Advantages of Blazor Server

✔ Smaller Download Size – The app loads quickly because it doesn’t need to download .NET runtime and dependencies.
✔ Leverages Server Resources – Uses server computing power, ideal for complex applications.
✔ Secure Data Handling – Data is processed on the server, reducing exposure to client-side attacks.
✔ Works on All Browsers – No WebAssembly support required.

Disadvantages of Blazor Server

✖ Higher Latency – UI updates depend on server round trips, which can be slow for users with poor connectivity.
✖ Requires Persistent Connection – Needs a real-time connection via SignalR, which may be challenging in low-bandwidth environments.
✖ Server Load Increases with Users – Each user connection consumes server memory.

Blazor WebAssembly: Running .NET Code in the Browser

Blazor WebAssembly applications run entirely in the browser, executing .NET code natively using WebAssembly.

How Blazor WebAssembly Works

When a user loads the page, the entire application (C# code, .NET runtime, dependencies) is downloaded to the browser. 
The .NET code executes inside the browser’s WebAssembly engine, independent of the server. 
Any server communication happens via HTTP API calls, like a traditional frontend app. 
The UI is rendered and updated client-side, without requiring server-side interaction.

✔ Client-Side Execution – The app runs independently of the server, reducing server load.
✔ Works Offline – Once loaded, the application can function without an internet connection.
✔ Full Control Over UI Rendering – No latency issues from server round trips.
✔ Scalability – Since processing happens on the client, the server load remains low.

Disadvantages of Blazor WebAssembly

✖ Larger Initial Load Time – The browser must download .NET runtime, libraries, and app dependencies before running.
✖ Limited Browser Capabilities – Can’t interact directly with file systems, system resources, or privileged APIs.
✖ Performance Considerations – Performance depends on the browser's WebAssembly execution speed.

7.1.3 When to Use Blazor Server vs. Blazor WebAssembly

Both Blazor Server and Blazor WebAssembly offer powerful capabilities, but choosing between them depends on your application's needs.

Use Blazor Server When:

● Users have stable network connections (e.g., enterprise applications on a corporate network).

● Fast initial load times are critical.

● Server-side security and logic are preferred over client-side execution.

Use Blazor WebAssembly When:

● Low latency and high interactivity are required (e.g., gaming, real-time dashboards).

● The application needs to work offline or in low-connectivity environments.

● The server should not bear additional processing load (e.g., SaaS applications).

7.3 Using Dependency Injection in Blazor Projects

Dependency Injection (DI) is a core design pattern in ASP.NET Core and Blazor that helps manage dependencies, reduce tight coupling, and improve testability. In Blazor, DI allows components and services to be injected efficiently, making the code more modular and maintainable.

This section explores how dependency injection works in Blazor, how to register and inject services, and how to use DI in both Blazor Server and Blazor WebAssembly applications.

7.3.1 What is Dependency Injection?

Dependency Injection (DI) is a technique that manages the creation and lifetime of objects and their dependencies. Instead of a class creating its own dependencies, DI injects them from an external container.

Why Use Dependency Injection in Blazor?

● Encapsulates logic in reusable services, keeping components clean.

● Improves testability by making it easy to mock services.

● Enhances maintainability by managing dependencies centrally.

● Optimizes performance by controlling object lifetimes efficiently.

In Blazor, DI is built into the framework and follows the same principles as ASP.NET Core.

7.3.2 Setting Up Dependency Injection in Blazor

To use DI in a Blazor project, you need to register services in Program.cs and inject them into components.

Step 1: Register Services in Program.cs

Modify Program.cs to register a service:

var builder = WebApplication.CreateBuilder(args);

Step 2: Create a Service (WeatherService.cs)

public class WeatherService

{

private static readonly string[] Summaries = new[]

{

"Sunny", "Cloudy", "Rainy", "Stormy", "Windy", "Snowy"

};

public Task<string> GetWeatherAsync()

{

var random = new Random();

var forecast = Summaries[random.Next(Summaries.Length)];

return Task.FromResult(forecast);

}

● This service generates random weather forecasts asynchronously.

● It will be injected into a Blazor component.

Step 3: Inject the Service in a Blazor Component (Weather.razor)

}

Explanation:

● The @inject directive injects WeatherService into the Blazor component.

● Clicking the button calls GetWeather(), which fetches a random weather forecast.

● DI keeps the component clean by separating logic into a service.

7.3.3 Dependency Injection Scopes in Blazor

DI services have different lifetimes in Blazor:

Singleton – One instance shared across the application. 
Scoped – A new instance per user session. 
Transient – A new instance per request. 

Registering Different Service Lifetimes

Modify Program.cs:

When to Use Each Lifetime?

● Use Singleton for global data like app configurations.

● Use Scoped for user-specific data in Blazor Server.

● Use Transient for stateless operations like logging.

8.1 Performance Tuning for .NET 9 Applications

Performance optimization is crucial in ensuring that .NET 9 applications run efficiently, use resources effectively, and scale without bottlenecks. Whether you're building a web API, desktop application, or microservices, applying structured performance tuning techniques helps reduce memory usage, CPU overhead, and response times.

This section provides practical strategies for tuning .NET 9 applications, covering garbage collection, memory management, async programming, database performance, and profiling tools.

8.1.1 Understanding Performance Bottlenecks in .NET 9

Every application has potential performance bottlenecks, and they typically fall into these categories:

CPU Bottlenecks – Excessive CPU usage due to inefficient algorithms, unnecessary computations, or high thread contention. 
Memory Leaks – Objects retained in memory unnecessarily, leading to excessive garbage collection. 
Slow I/O Operations – Blocking network calls, disk reads/writes, or database queries causing delays. 
Inefficient Concurrency – Poor thread management leading to thread starvation or excessive context switching. 
Overhead from Unoptimized Code – Redundant loops, unnecessary LINQ operations, or excessive object allocations. 

The key to optimizing performance is to measure before making changes, ensuring improvements are data-driven and not guesswork.

8.1.2 Profiling and Monitoring Performance in .NET 9

8.1.3 Optimizing Memory Usage in .NET 9

1. Avoiding Large Object Heap (LOH) Allocations

Objects larger than 85KB are allocated in the Large Object Heap (LOH), which increases garbage collection overhead. Avoid frequent large allocations by reusing buffers.

Example: Using ArrayPool<T> Instead of Allocating Large Arrays

using System.Buffers;

var pool = ArrayPool<byte>.Shared;

var buffer = pool.Rent(100_000); // Rent a buffer instead of allocating a new array

try

{

// Use buffer

}

finally

{

pool.Return(buffer); // Return the buffer for reuse

}

This reduces memory fragmentation and improves allocation performance.

2. Using Structs Instead of Classes for Small Data Types

Structs are stack-allocated, reducing pressure on the garbage collector.

Example: Converting a Small Class to a Struct

struct Point

{

public int X { get; set; }

public int Y { get; set; }

}

Use structs for small, immutable data types to avoid unnecessary heap allocations.

3. Implementing Object Pooling

If an application frequently creates and disposes of objects, it is more efficient to reuse them using object pooling.

Example: Using ObjectPool<T> for Reusing Objects

using Microsoft.Extensions.ObjectPool;

var provider = new DefaultObjectPoolProvider();

var pool = provider.Create<MyReusableObject>();

var obj = pool.Get();

// Use the object...

pool.Return(obj); // Return object to the pool

This reduces garbage collection overhead and improves performance.

8.1.4 Optimizing Asynchronous Code for Performance

1. Avoid Blocking Calls in Async Code

Avoid .Result and .Wait(), as they cause thread blocking and performance issues.

Bad Code (Blocking Thread):

var result = httpClient.GetStringAsync("https://api.example.com").Result; // Blocks the thread

Optimized Code (Asynchronous Execution):

var result = await httpClient.GetStringAsync("https://api.example.com"); // Non-blocking

Using await prevents thread blocking and improves responsiveness.

2. Use ValueTask<T> for Performance Optimization

For frequently called async methods, use ValueTask<T> instead of Task<T> to reduce memory allocations.

Example: Using ValueTask<T> Instead of Task<T>

public ValueTask<int> GetCountAsync()

{

return new ValueTask<int>(42); // Returns a precomputed value without allocation

}

This minimizes unnecessary heap allocations for short-lived async methods.

8.1.5 Improving Database Performance

1. Optimize Entity Framework Core Queries

Use .AsNoTracking() for read-only queries to avoid unnecessary change tracking.

Example: Optimized Query

var products = await _context.Products.AsNoTracking().ToListAsync();

This reduces memory usage and improves query performance.

2. Use Batching for Bulk Database Operations

Instead of sending multiple insert queries, use bulk operations.

Example: Batching Insert Statements

await _context.BulkInsertAsync(products);

This minimizes database round-trips, improving performance.

8.1.6 Reducing Startup Time

1. Enable ReadyToRun (R2R) Compilation

ReadyToRun (R2R) precompiles .NET assemblies, reducing startup time.

Enable R2R in .csproj:

<PropertyGroup>

<PublishReadyToRun>true</PublishReadyToRun>

</PropertyGroup>

2. Trim Unused Assemblies with dotnet trim

For smaller app sizes, trim unused dependencies:

dotnet publish -c Release -p:PublishTrimmed=true

This removes unnecessary assemblies, reducing application size and load time.

8.1.7 Best Practices for Performance Optimization

Use asynchronous programming (async/await) to prevent thread blocking. 
Optimize memory usage by reducing object allocations and using pooling. 
Use Structs over Classes for small data types to avoid heap allocations. 
Minimize database queries by caching frequently used data. 
Enable Response Compression in ASP.NET Core to reduce payload size. 
Monitor performance regularly using .NET counters and profiling tools. 
Use ReadyToRun and trimming for faster startup and smaller binaries. 

8.1.8 Conclusion

Performance tuning is not a one-time task—it requires continuous monitoring and data-driven optimizations. By applying memory optimizations, async best practices, database query tuning, and efficient object management, .NET 9 applications can run faster, consume fewer resources, and scale effectively.

8.3.3 Writing a Dockerfile for a .NET 9 Application

A Dockerfile is a script that contains instructions on how to build and run a containerized application.

1. Create a Simple .NET 9 Web API

If you don’t have a .NET 9 app yet, create a new one:

dotnet new webapi -n DockerizedApp

cd DockerizedApp

2. Create a Dockerfile in the Project Root

# Use the official .NET 9 runtime image

FROM mcr.microsoft.com/dotnet/aspnet:9.0 AS base

WORKDIR /app

EXPOSE 8080

# Use the .NET 9 SDK image to build the application

FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build

WORKDIR /src

COPY ["DockerizedApp.csproj", "./"]

RUN dotnet restore "./DockerizedApp.csproj"

COPY . .

RUN dotnet publish -c Release -o /app/publish

# Final stage: Run the application

FROM base AS final

WORKDIR /app

COPY --from=build /app/publish .

ENTRYPOINT ["dotnet", "DockerizedApp.dll"]

Explanation of the Dockerfile

● Stage 1 (Base Image): Uses the ASP.NET 9 runtime as a base.

● Stage 2 (Build & Publish): Copies the source code, restores dependencies, and builds the application.

● Stage 3 (Final Image): Runs the published application using the optimized runtime-only image.

8.3.4 Building and Running the Docker Container

1. Build the Docker Image

From the project root, run:

docker build -t dockerizedapp .

This creates a container image named dockerizedapp.

2. Run the Container

Start the container using:

docker run -d -p 8080:8080 --name myapp dockerizedapp

● -d → Runs the container in the background (detached mode).

● -p 8080:8080 → Maps container port 8080 to host port 8080.

● --name myapp → Assigns the container a human-readable name.

Now, access the running .NET 9 API in your browser:

http://localhost:8080/swagger

8.3.5 Optimizing .NET 9 Docker Images

Docker images should be optimized to reduce size, build time, and security risks.

1. Use Multi-Stage Builds

Multi-stage builds minimize final image size by excluding unnecessary files.

Instead of:

Use separate build and runtime stages:

FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build

2. Enable Ready-to-Run (R2R) Compilation

Improve startup time by precompiling native code.

Modify the Dockerfile:

RUN dotnet publish -c Release -r linux-x64 --self-contained true /p:PublishReadyToRun=true -o /app/publish

3. Use Alpine-based Images for Smaller Size

Instead of:

FROM mcr.microsoft.com/dotnet/aspnet:9.0

Use Alpine-based images:

FROM mcr.microsoft.com/dotnet/aspnet:9.0-alpine

This reduces image size significantly.

9.1.3 Preventing SQL Injection Attacks

SQL Injection occurs when malicious SQL commands are injected into queries, allowing attackers to manipulate databases.

9.2 Implementing Role-Based and Policy-Based Authorization

Authorization ensures that users only access the resources and functionalities they are permitted to use. While authentication verifies who a user is, authorization determines what a user is allowed to do.

.NET 9 provides two primary approaches to authorization:

● Role-Based Authorization – Grants access based on predefined user roles.

● Policy-Based Authorization – Uses custom policies and conditions to enforce fine-grained access control.

This section explores how to configure, implement, and enforce both authorization strategies using .NET 9.

9.2.1 Understanding Role-Based Authorization

Role-based authorization assigns permissions based on user roles. For example:

● Admin users can access everything.

● Managers can view and edit specific sections.

● Users can only view their own data.

This approach is suitable for applications with predefined roles, such as internal enterprise applications.

9.2.2 Setting Up Role-Based Authorization in .NET 9

Step 1: Enable Authentication and Authorization in Program.cs

using Microsoft.AspNetCore.Authentication.JwtBearer;

using Microsoft.AspNetCore.Authorization;

using Microsoft.AspNetCore.Identity;

using Microsoft.IdentityModel.Tokens;

using System.Text;

var builder = WebApplication.CreateBuilder(args);

// Configure Authentication (JWT-based)

builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)

.AddJwtBearer(options =>

{

options.TokenValidationParameters = new TokenValidationParameters

{

ValidateIssuer = true,

ValidateAudience = true,

ValidateLifetime = true,

ValidateIssuerSigningKey = true,

ValidIssuer = "https://your-issuer.com",

ValidAudience = "your-audience",

IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes("Your_Secret_Key"))

};

});

// Add Authorization Services

builder.Services.AddAuthorization(options =>

{

options.AddPolicy("AdminOnly", policy => policy.RequireRole("Admin"));

});

var app = builder.Build();

app.UseAuthentication();

app.UseAuthorization();

app.Run();

Step 2: Assign Roles to Users in Identity System

When registering users, assign them specific roles:

var user = new ApplicationUser { UserName = "admin@example.com", Email = "admin@example.com" };

await userManager.CreateAsync(user, "SecurePassword123!");

await userManager.AddToRoleAsync(user, "Admin");

Step 3: Enforce Role-Based Access Control in Controllers

Modify your API controller to restrict access based on roles.

using Microsoft.AspNetCore.Authorization;

using Microsoft.AspNetCore.Mvc;

[ApiController]

[Route("api/admin")]

public class AdminController : ControllerBase

{

[Authorize(Roles = "Admin")]

[HttpGet]

public IActionResult GetAdminData()

{

return Ok(new { Message = "Only Admins can access this data." });

}

How This Works:

● The [Authorize(Roles = "Admin")] attribute ensures only users with the "Admin" role can access the endpoint.

● Unauthorized users receive 403 Forbidden responses.

Step 4: Enforce Multiple Roles for an Endpoint

You can allow multiple roles for a specific API:

[Authorize(Roles = "Admin,Manager")]

[HttpGet]

public IActionResult GetManagerData()

{

return Ok(new { Message = "Admins and Managers can access this." });

}

Now, both "Admin" and "Manager" roles can access this API.

9.2.3 Understanding Policy-Based Authorization

Role-based authorization may not be enough in complex applications. Policy-based authorization allows defining custom rules beyond roles.

For example, instead of just checking if a user is an Admin, you can check:

● Account Age (e.g., Users with accounts older than 1 year).

● Department-Based Access (e.g., Only employees in the HR department).

● Custom Security Clearance Levels.

Use policy-based authorization when:
✔ You need fine-grained access control beyond predefined roles.
✔ Access should be determined by multiple conditions.

9.2.4 Implementing Policy-Based Authorization in .NET 9

Step 1: Define Policies in Program.cs

Modify the Authorization services to add a custom policy.

builder.Services.AddAuthorization(options =>

{

options.AddPolicy("MustBeEmployee", policy =>

policy.RequireClaim("Department", "HR", "IT", "Finance"));

options.AddPolicy("SeniorUsersOnly", policy =>

policy.RequireAssertion(context =>

context.User.HasClaim(c => c.Type == "AccountAge" && int.Parse(c.Value) > 365)));

});

Step 2: Assign Claims to Users

Claims are key-value pairs stored inside authentication tokens.

When creating a user, add custom claims:

var claims = new List<Claim>

{

new Claim("Department", "IT"),

new Claim("AccountAge", "400")

};

var claimsIdentity = new ClaimsIdentity(claims, "Custom");

var claimsPrincipal = new ClaimsPrincipal(claimsIdentity);

Now, the user has custom attributes (Department, AccountAge) stored in their authentication token.

Step 3: Enforce Policy-Based Authorization in Controllers

Use the [Authorize(Policy = "MustBeEmployee")] attribute to restrict access.

[Authorize(Policy = "MustBeEmployee")]

[HttpGet]

public IActionResult GetEmployeeData()

{

return Ok(new { Message = "Only employees in HR, IT, or Finance can access this." });

}

For the AccountAge policy:

[Authorize(Policy = "SeniorUsersOnly")]

[HttpGet]

public IActionResult GetSeniorUsers()

{

return Ok(new { Message = "Only users with an account older than 1 year can access this." });

}

Now, users must meet the policy criteria to access these APIs.

9.2.5 Creating Custom Authorization Handlers

For even more advanced scenarios, you can create custom authorization handlers.

Step 1: Create a Custom Authorization Requirement

using Microsoft.AspNetCore.Authorization;

public class MinimumExperienceRequirement : IAuthorizationRequirement

{

public int MinimumYears { get; }

public MinimumExperienceRequirement(int years)

{

MinimumYears = years;

}

Step 2: Create an Authorization Handler

using Microsoft.AspNetCore.Authorization;

using System.Security.Claims;

using System.Threading.Tasks;

public class MinimumExperienceHandler : AuthorizationHandler<MinimumExperienceRequirement>

{

protected override Task HandleRequirementAsync(AuthorizationHandlerContext context, MinimumExperienceRequirement requirement)

{

var experienceClaim = context.User.FindFirst("ExperienceYears");

if (experienceClaim != null && int.Parse(experienceClaim.Value) >= requirement.MinimumYears)

{

context.Succeed(requirement);

}

return Task.CompletedTask;

}

Step 3: Register the Policy in Program.cs

builder.Services.AddSingleton<IAuthorizationHandler, MinimumExperienceHandler>();

builder.Services.AddAuthorization(options =>

{

options.AddPolicy("ExperiencedUsersOnly", policy =>

policy.Requirements.Add(new MinimumExperienceRequirement(5)));

});

Step 4: Apply the Custom Policy in a Controller

[Authorize(Policy = "ExperiencedUsersOnly")]

[HttpGet]

public IActionResult GetExperiencedUsers()

{

return Ok(new { Message = "Only users with 5+ years of experience can access this." });

}

10.2 Building Microservices with .NET 9 and Kubernetes

Microservices architecture is a modern approach to building scalable, flexible, and maintainable applications. Instead of a monolithic structure, where all components are tightly coupled, microservices break applications into independent services that communicate through well-defined APIs.

.NET 9 is designed to support microservices development, offering powerful tools for containerization, orchestration, communication, and scalability. When combined with Kubernetes, microservices can be deployed and managed efficiently across cloud and on-premises environments.

This section covers the fundamentals of microservices with .NET 9, including building a microservice, containerizing it with Docker, deploying it to Kubernetes, and implementing inter-service communication.

10.2.1 Understanding Microservices Architecture

Microservices follow a distributed system approach, where an application is composed of independent services.

Key Characteristics of Microservices

✔ Independence – Each service operates independently and can be deployed, updated, or scaled separately.
✔ Loose Coupling – Services communicate through APIs, not direct dependencies.
✔ Polyglot Support – Services can be built using different programming languages and databases.
✔ Scalability – Individual services scale based on demand instead of scaling the entire system.

Example: E-Commerce Microservices System

A monolithic e-commerce app might have:

● User Management

● Orders Processing

● Product Catalog

● Payments Service

In a microservices architecture, these features are split into independent services, each with its own database and API.

UserService → Manages user accounts and authentication 

ProductService → Handles product catalog and inventory 

OrderService → Processes customer orders 

PaymentService → Manages payments and transactions

Each microservice runs as an independent service and communicates via HTTP or message queues (RabbitMQ, Kafka, etc.).

10.2.2 Building a Microservice in .NET 9

We will build a Product Microservice using .NET 9 and ASP.NET Core Web API.

1. Create a New Microservice Project

10.3 Migrating Legacy Applications to .NET 9

Migrating legacy applications to .NET 9 offers improved performance, security, and maintainability. Legacy systems often run on older .NET Framework versions, which lack modern features, cloud support, and security enhancements. However, migration is not always straightforward. It requires careful planning, understanding dependencies, and ensuring compatibility.

This section covers why migration is necessary, different migration strategies, practical steps for upgrading to .NET 9, and how to handle common challenges.

10.3.1 Why Migrate to .NET 9?

Many organizations still use .NET Framework (4.x) or .NET Core (2.x, 3.x), which have limitations compared to .NET 9. Migrating brings several benefits:

✔ Performance Improvements – .NET 9 includes Just-In-Time (JIT) compiler optimizations, enhanced garbage collection, and faster API processing.
✔ Cross-Platform Compatibility – Run applications on Windows, Linux, and macOS without modification.
✔ Cloud-Native Support – .NET 9 is optimized for Azure, AWS, and Kubernetes, enabling serverless and microservices architectures.
✔ Security Enhancements – Built-in TLS 1.3, stronger cryptography, and faster authentication mechanisms.
✔ Long-Term Support (LTS) – Future-proof your application with continued updates and official Microsoft support.

If your application relies on outdated .NET versions, it risks security vulnerabilities, lack of support, and compatibility issues with modern infrastructure.

10.3.2 Migration Strategies

Migration is not a one-size-fits-all process. Depending on the complexity of your application, choose one of the following approaches:

1. Lift-and-Shift (Rehosting)

● Move the existing application as-is to a .NET 9 runtime without refactoring.

● Works best for ASP.NET MVC/WebForms apps hosted on IIS or Azure App Services.

● Pros: Fastest approach, minimal code changes.

● Cons: Doesn't leverage .NET 9 features or optimizations.

2. Incremental Upgrade (Phased Approach)

● Upgrade the application in stages, migrating specific services or modules to .NET 9 gradually.

● Useful for large enterprise applications with multiple dependencies.

● Pros: Less risk, easier rollback if issues occur.

● Cons: Requires maintaining both old and new codebases temporarily.

3. Full Replatforming (Rewriting in .NET 9)

● Rewrite the application from scratch, using modern architectures like microservices, Blazor, or gRPC.

● Pros: Best for long-term sustainability and cloud scalability.

● Cons: Time-consuming and requires extensive testing.

Choose the strategy based on your application's complexity, business goals, and resource availability.

10.3.3 Preparing for Migration

Before starting migration, perform the following steps:

1. Assess the Existing Application

Use the .NET Upgrade Assistant to analyze compatibility:

dotnet tool install --global upgrade-assistant

upgrade-assistant analyze MyLegacyApp.sln

This tool identifies: ✔ Outdated APIs
✔ Unsupported dependencies
✔ Compatibility issues with .NET 9

2. Identify Third-Party Dependencies

● Check if external libraries are compatible with .NET 9.

● Use NuGet Package Manager to update dependencies:

dotnet list package --outdated

dotnet add package [PackageName] --version latest

● If a library isn't supported, consider alternatives or custom implementations.

3. Backup and Version Control

● Store the current codebase in a Git repository before migration.

● If issues occur, rollback is easy:

git checkout -b migration-backup
